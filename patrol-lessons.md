# BugHunt 巡检：实现方式与踩坑记录

## 巡检架构

```
cron (每15分钟)
  → bughunt-patrol.py
    → git pull (同步 repo 最新结果)
    → 扫描 results/worker-XX/*.json (自动更新状态)
    → 4 条规则逐个检查
    → 汇总 → 广播群
    → 异常 → 对应 1v1 群
    → 更新 patrol-status.json
```

### 关键文件

| 文件 | 位置 | 作用 |
|------|------|------|
| 巡检脚本 | PM 本机 `bughunt-patrol.py` | cron 触发，执行巡检逻辑 |
| 状态文件 | PM 本机 `bughunt-patrol-status.json` | 存储 Worker 进度、信号时间、通道配置 |
| 巡检日志 | PM 本机 `bughunt-patrol.log` | cron 输出记录 |
| 巡检方案 | repo `patrol-plan.md` | 规则设计文档 |

### 4 条巡检规则

1. **失联检测** — 最后信号超 20 分钟 → 1v1 群询问
2. **连续失败** — 同一 Worker 连续 3 个 failed → 暂停 + 要求诊断
3. **速率预警** — 实际完成速率 < 目标 80% → 广播群预警
4. **质量预警** — unclear 占比 > 30% → 抽查原因

### 数据源：repo 自动同步

巡检脚本每次执行时先 `git pull`，扫描 `results/worker-XX/` 目录下所有 JSON 文件，自动统计 completed/failed/normal/abnormal/unclear，写入 patrol-status.json。不依赖 Worker 手动上报状态。

## 踩过的坑

### 坑 1：null 信号检测 bug

**现象：** Fabrice 从未发过状态信号，`last_signal_time` 为 null，但巡检一直没标他失联。

**根因：** 失联检测代码是 `if last_signal:` 走计算逻辑，else 什么都不做。null 走了 else 分支，直接跳过，永远不触发失联。

**修复：** else 分支加判断——已分配任务但 `last_signal_time` 为 null → 标为"从未上线"，触发失联规则。

**教训：** 边界值（null/0/空字符串）必须作为测试用例覆盖，不能只测正常路径。

---

### 坑 2：速率预警规则完全没实现

**现象：** 巡检方案里写了 4 条规则，但实际代码只实现了 3 条，规则 3（速率预警）漏掉了。

**根因：** 写代码时按规则 1、2 顺序写，写完 2 直接跳到了 4（质量预警），中间忘了 3。没有对照设计文档逐条检查。

**修复：** 补上速率预警逻辑——计算实际完成速率 vs 目标速率，低于 80% 触发。

**教训：** 实现完必须对照设计文档逐条 checklist，不能凭记忆。

---

### 坑 3：状态更新断裂——手动维护不可持续

**现象：** patrol-status.json 最初设计为 PM 手动更新（每次收到 Worker 信号就改 JSON），实际运行中 PM 很快跟不上，状态数据过时。

**根因：** Worker 执行速度快，信号密集，PM 手动更新 JSON 是串行瓶颈。

**修复：** 新增 `sync_from_repo()` 函数——每次巡检先 `git pull`，扫描 `results/` 目录自动统计。完全去掉手动更新依赖。

**教训：** 巡检数据源必须自动化。任何依赖人工手动更新的环节，在执行压力下一定会断。

---

### 坑 4：@ mention 显示名用了内部标识

**现象：** 巡检异常通知发到 1v1 群时，消息里写的是 `@worker-fabrice`、`@worker-moss`，而不是 `@Fabrice`、`@Moss`。

**根因：** 代码里直接用了 status.json 的 key（`worker-fabrice`）拼接 @ 消息，没有做显示名映射。

**修复：** 新增 `DISPLAY_NAMES` 字典，`worker-fabrice` → `Fabrice`，`worker-moss` → `Moss`。

**教训：** 内部标识和面向用户的显示名要分开，代码里用 key，消息里用显示名。

---

### 坑 5：巡检消息进了主群而非分区

**现象：** 广播群有分区（`2043640961514344448`），巡检消息发到了主群而不是分区。

**根因：** DMWork bot API 发消息时 `channel_id` 指向主群，没有指定分区的参数。目前不清楚 bot API 是否支持分区定向发送。

**状态：** 未解决。POC 阶段林菡确认发主群可接受，后续需研究分区发送方式。

---

### 坑 6：bot API @ mention 不触发通知

**现象：** 消息 content 里写了 `@Moss`，payload 里也加了 `mention.uids`，但对方没收到 @ 通知。

**根因：** 待确认。可能是 bot 发消息和人类发消息的 @ 机制不同。

**状态：** 未解决。Worker 目前靠轮询群消息而非 @ 通知来获取指令。

---

## 可复用的经验

1. **巡检数据源必须自动化** — 不依赖人手动更新状态文件
2. **边界值必须测试** — null、0、空字符串、从未上线
3. **实现后对照设计文档逐条 checklist** — 不能凭记忆
4. **内部标识 ≠ 显示名** — 面向用户的消息要做映射
5. **先验证通道再上线** — 巡检部署前先手动测试每个通道（广播群、1v1 群）是否通达
6. **状态文件保持最小** — 只存无法从 repo 推导的数据（如通道配置），可推导的（进度、统计）每次从源头重算
