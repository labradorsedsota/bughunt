# BugHunt — CUA 轨迹数据采集（开源 Bug 复现方案）

## 一、项目定位

**项目代号：** BugHunt

**一句话目标：** 从 GitHub 开源项目的已修复 UI Bug 中大规模采集 mano-cua 操作轨迹数据，用于 CUA 模型训练和评测。

**核心逻辑：** 已修复的 Bug = ground truth（已知答案的考试）。checkout 到修复前的 buggy 版本 → 部署 → mano-cua 盲测 → 记录操作轨迹 + 是否命中 Bug。

**数据需求方：** 老傅（FTY），CUA 模型负责人。

### 与自研应用方案的区别

CUA 轨迹数据的采集有两条路径：

| 维度 | 自研应用方案（之前在做的） | 开源 Bug 复现方案（本项目） |
|------|------------------------|--------------------------|
| 应用来源 | 自己从需求到开发到部署 | GitHub 开源项目直接拿来用 |
| 测试用例来源 | 自己编写 test case | 从已修复的 Bug Issue 中提取功能点 |
| Ground Truth | 无（需要人工判断） | 有（fix PR = 答案） |
| 规模化速度 | 慢（每个应用都要走完整流程） | 快（筛选 + 部署即可批量跑） |
| 数据多样性 | 受限于自研应用的类型 | 覆盖各种技术栈、组件类型、Bug 类型 |
| 终点 | 相同：mano-cua 操作轨迹数据 | 相同 |

两条路径互补，不是替代关系。本项目聚焦开源 Bug 复现方案。

## 二、全链路流程

```
Phase 1: 筛选准备
  ① 筛选项目（Mac 可跑 + UI Bug 多 + 组件丰富）
  ② 筛选 Bug（GUI 可观测 + fix commit 可追溯）
  ③ 构造任务清单（中文任务描述，不暴露 Bug）

Phase 2: 批量执行
  ④ 部署 buggy 版本（checkout fix PR 的 parent commit）
  ⑤ 执行 mano-cua 测试
  ⑥ 收集结果（sess-id、是否复现、步数）

Phase 3: 汇总交付
  ⑦ 汇总报告
  ⑧ 交付老傅
```

### 筛选标准（老傅已确认）

**项目筛选：**
1. Mac 上能直接跑起来（clone → 简单命令 → 浏览器可访问）
2. PR 列表中基于 UI 界面测出的 Bug 数量足够多
3. 测试用例足够多，界面覆盖的组件类型足够丰富（表单/表格/导航/弹窗/轮播/侧边栏等）

**Bug 筛选：**
4. Bug 必须是 GUI 可观测的（排除纯后端/性能/并发类）
5. Fix commit 可追溯，能 checkout 到修复前代码
6. 不排除动态过程类 Bug（动画/过渡等）

### 产出格式（老傅已确认）

```
项目名称 | 测试内容 | sess-id | 是否复现
```

### 任务描述规则
- 中文描述
- 只说要验证的功能点，**不暴露 Bug 是什么**
- 例：「打开推荐页，点击翻页按钮逐页浏览，检查切换过程是否正常流畅」

## 四、执行 Agent 编排方案

### 4.1 架构：1 调度 + N 执行

```
                    ┌─────────────┐
                    │   Pichai    │ ← 项目管理 & 调度
                    │  (PM/调度)   │
                    └──────┬──────┘
                           │ 任务分发 / 进度汇总
              ┌────────────┼────────────┐
              │            │            │
        ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐
        │ Worker 1  │ │  ...  │ │ Worker 10 │
        │ (执行Agent)│ │       │ │ (执行Agent)│
        └───────────┘ └───────┘ └───────────┘
              │                       │
              └───────┬───────────────┘
                      ↓
              智子（质量抽检 & 汇总协助）
```

### 4.2 任务包格式

每个 Worker 接收一组任务包：

```yaml
task_id: "BH-001"
project: "ivegamsft/luxesite"
bug_issue: "#258"
buggy_commit: "39a6e11"
deploy_command: "pnpm install && pnpm dev"
test_description: "打开推荐页，点击翻页按钮逐页浏览，检查切换过程是否正常流畅"
expected_url: "http://localhost:3000"
```

### 4.3 Worker 执行流程

1. Clone 项目 → checkout buggy_commit
2. 执行 deploy_command → 确认服务启动
3. 调用 mano-cua 执行测试（传入 test_description + URL）
4. 收集结果：sess-id、是否复现、步数
5. 上报结果到 PM

### 4.4 任务分配策略

**按项目分组：** 同一个项目的多个 Bug 分配给同一个 Worker。
- 避免重复 clone/部署
- Worker 对项目越熟悉执行越稳定
- 单个项目部署出问题不阻塞其他 Worker

### 4.5 通信机制

| 阶段 | 方式 | 说明 |
|------|------|------|
| 起步验证 | DMWork 群消息 | 可追溯、人类可观测 |
| 规模化执行 | 文件 + API 驱动 | 结构化、可并行、不依赖消息 |

建议先用 DMWork 群跑通流程，再切结构化方案跑量。

## 五、质量控制

### 5.1 数据质量标准

| 维度 | 标准 |
|------|------|
| 轨迹完整性 | mano-cua session 正常结束（非中断/超时） |
| 任务描述质量 | 中文、不暴露 bug、只描述要验证的功能点 |
| 复现判定 | 明确标注：✅ 复现 / ⚠️ 部分复现 / ❌ 未复现 |
| 产出格式 | 项目名称 \| 测试内容 \| sess-id \| 是否复现 |

### 5.2 PM 质量把控

1. **首轮抽检**：每个 Worker 的前 3 个 case，PM 检查轨迹质量
2. **异常监控**：超时率 > 30% 或未复现率 > 70% 的 Worker 暂停排查
3. **每日汇总**：每天向老傅 & Emily 同步进度
